# Changelog

## v0.1.0 - 2026-05-30

First tagged release. The library is correctness-verified against the
independent `golang.org/x/crypto/ripemd160` oracle and ships a real, faster
SIMD backend on arm64.

### API

- Public `ripemd160mb` API: `Hash32` (multi-buffer fixed-32 fast path),
  `HashEach`, `New`, `Sum`, `Lanes`, and `Backend`.
- Optional `hash160` subpackage with batched `RIPEMD160(SHA256(x))`.

### Backends

- Hand-tuned 4-lane arm64 NEON kernel, generated by `internal/neongen` and
  verified bit-for-bit against the scalar oracle. It is **~3x faster than
  scalar** on Apple M3 (about 13.2M vs 4.3M hashes/s at `n = 2048`,
  `GOMAXPROCS=1`) and allocates nothing. It is the default on arm64.
- Portable pure-Go scalar backend and streaming `hash.Hash`, the default off
  arm64 and the correctness oracle everywhere.
- `Backend()`/`Lanes()` always report the kernel that actually runs; the
  library never advertises a SIMD backend that secretly falls back to scalar.
- amd64 SIMD (SSE2/AVX2/AVX-512) is not yet implemented and runs scalar; the
  NEON generator is the reference template for adding it.

### Testing

- Differential tests and fuzzing (`FuzzSum`, `FuzzHash32`, `FuzzHash160_32`)
  against the `x/crypto` reference; every available backend is exercised across
  the lane/tail boundary matrix.
- Contract tests: `Hash32` is zero-allocation on every backend, is safe for use
  by many concurrent workers, and transparently handles message counts that are
  not a multiple of the lane width.
- Tests pass under `-race` and under forced-scalar execution. Library coverage
  (root + `hash160`) is 100%.

### Notable fixes from the initial draft

- The NEON kernel generator previously emitted assembly that could not even
  build (`VLD1R` with a memory displacement) and had clobbering bugs in the
  message transpose, the digest store, and the finalization. The kernel is now
  correct and tuned (constants kept in registers, per-round constant
  materialization, zero padding-word skips).
- The NEON generator wrote its output two directories above the repository root
  because of a working-directory mismatch, so `go generate` never actually
  refreshed the committed assembly. Output path resolution is fixed and now
  asserts it resolved to this module before writing.
- Removed the unimplemented amd64 SSE2/AVX2/AVX-512 backend stubs and their
  generator, which advertised 4/8/16 lanes while running the scalar fallback.
- Dropped the unused `golang.org/x/sys` dependency.
